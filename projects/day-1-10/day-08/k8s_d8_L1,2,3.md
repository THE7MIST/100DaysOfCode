# Kubernetes Fundamentals and YAML Lab Notes

## Kubernetes Core Concepts

### Container Orchestration Platforms

Container orchestration platforms automate the deployment, management, scaling, networking, and recovery of containers across multiple systems.

Common orchestration platforms:

* Docker Swarm
* Kubernetes
* Apache Mesos

### Why Container Orchestration is Needed

Managing a small number of containers manually is feasible:

```bash
docker run nginx
docker run redis
docker run mysql
```

However, managing hundreds or thousands of containers requires automation.

Container orchestration provides:

* Automated deployment
* Automatic scaling
* Self-healing
* Load balancing
* Scheduling
* Resource management

### Docker Swarm

Docker's native orchestration platform.

Advantages:

* Easy to use
* Lightweight
* Integrated with Docker

Limitations:

* Smaller ecosystem
* Less feature-rich than Kubernetes

### Kubernetes

Kubernetes is the most widely used container orchestration platform.

Originally developed by Google.

Common users include:

* Google
* Netflix
* Spotify
* Adobe
* IBM
* AWS
* Microsoft Azure

Capabilities:

* Thousands of nodes
* Millions of containers
* Automated deployment
* Automated scaling
* Service discovery
* Self-healing
* Load balancing

### Apache Mesos

Apache Mesos is a distributed systems platform originally designed for:

* Large-scale distributed systems
* Big data workloads

Mesos is less common today compared to Kubernetes.

```text
Users
   ↓
Orchestration Layer
   ↓
```

Example:


* Container placement

# Pods

## Definition

A Pod is the smallest deployable unit in Kubernetes.

Kubernetes manages Pods, not containers directly.

Relationship:

```text
Container
    ↓
Inside
    ↓
Pod
```

Analogy:

```text
House = Pod
Person = Container
```

Kubernetes manages houses, while containers reside inside them.

---

# Why Pods Exist

Pods provide:

* Shared networking
* Shared storage
* Shared lifecycle
* Scheduling abstraction

## Self-Healing

When a pod fails:

```text
Pod Crashes
      ↓
Kubernetes Detects Failure
      ↓
Pod Recreated
```

This capability is called:

**Self-Healing**

---

# Desired State Management

Users define the desired state.

Example:

```text
Desired Pods = 3
Current Pods = 2
```

Kubernetes automatically creates:

```text
1 Additional Pod
```

to match the desired state.

---

# Multi-Container Pods

A pod can contain one or more containers.

Example:

```text
Application Container
+
Logging Container
```

or

```text
Nginx
+
Monitoring Agent
```

This pattern is called:

**Sidecar Pattern**

---

# Helper Containers

Containers within the same pod share:

## Networking

Shared:

* IP Address
* localhost

Example:

Container A:

```text
localhost:5000
```

Container B can access:

```text
localhost:5000
```

without external networking.

## Storage

Containers can share volumes.

Example:

```text
Application writes logs
Helper container reads logs
```


### Step 1

Request sent to API Server.

### Step 2

Scheduler selects a node.

Pod created.

### Step 4

Container created.

### Step 5

Application starts.

Verify:

```bash
kubectl get pods
```

Example Output:

```text

```

## Q1. What is Container Orchestration?

Automation of deployment, scaling, networking, and management of containers across multiple hosts.

---

## Q2. Why Kubernetes instead of Docker alone?

Docker creates containers.

Kubernetes manages large numbers of containers automatically.

The smallest deployable unit in Kubernetes containing one or more containers.

## Q4. Can a Pod contain multiple containers?

Yes. Containers inside a pod share networking and storage.

Pods provide:

* Shared networking
* Shared storage

* Scheduling abstraction

---

## Q6. What happens when a Pod crashes?

Kubernetes automatically creates a replacement pod to maintain the desired state.

# Kubernetes Lab 3: YAML

## What is YAML?

YAML stands for:

**YAML Ain't Markup Language**

A human-readable data serialization format commonly used in Kubernetes manifests.

Typical Kubernetes structure:

```yaml
apiVersion:
kind:
metadata:
spec:
```

Benefits:

* Easy to read
* Less verbose than XML
* Cleaner than JSON
* Supports nested structures naturally

---

# YAML vs XML vs JSON

## XML

```xml
<Servers>
  <Server>
    <name>Server1</name>
    <owner>John</owner>
    <created>12232012</created>
    <status>active</status>
  </Server>
</Servers>
```

Characteristics:

* Uses tags
* Opening and closing tags required
* Highly verbose

---

## JSON

```json
{
  "Servers": [
    {
      "name":"Server1",
      "owner":"John",
      "created":12232012,
      "status":"active"
    }
  ]
}
```

Characteristics:

* Key-value pairs
* Uses braces and brackets
* Common in APIs

---

## YAML

```yaml
Servers:
  - name: Server1
    owner: John
    created: 12232012
    status: active
```

Characteristics:

* No braces
* No tags
* Uses indentation
* Highly readable

---

# YAML Building Blocks

## Key Value Pair

```yaml
name: apple
color: red
weight: 90g
```

## Array / List

```yaml
Fruits:
  - Apple
  - Banana
```

## Dictionary / Map

```yaml
employee:
  name: john
  gender: male
  age: 24
```

## List of Dictionaries

```yaml
Fruits:
  - Banana:
      Calories: 105
      Fat: 0.4g
      Carbs: 27g

  - Grape:
      Calories: 62
      Fat: 0.3g
      Carbs: 16g
```

---

# YAML Indentation Rules

Correct:

```yaml
employee:
  name: john
  age: 24
```

Incorrect:

```yaml
employee:
name: john
 age: 24
```

Use spaces only.

# Kubernetes YAML Keywords

## apiVersion

Specifies Kubernetes API version.

```yaml
apiVersion: v1
```

## kind

Defines resource type.

Examples:

```yaml
kind: Pod
kind: Service
kind: Deployment
kind: ConfigMap
```

## metadata

Stores object information.

```yaml
metadata:
  name: nginx
```

## spec

Defines desired state.

```yaml
spec:
```

Contains:

* Containers
* Ports
* Replicas
* Volumes
